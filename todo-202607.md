# MiniInflux.Net10 2026-07 性能优化清单

更新时间：2026-06-28

本文档基于 README 中 `2026-06-28 压测对比` 整理。当前核心差距是查询，其次是写入：

```text
Write throughput: MiniInflux 30,007.44 pts/s, InfluxDB 1.7.9 139,590.99 pts/s, gap 4.65x
Aggregate query latency: MiniInflux 175.62 ms, InfluxDB 1.7.9 1.92 ms, gap 91.70x
Raw LIMIT 1000 query latency: MiniInflux 47.47 ms, InfluxDB 1.7.9 7.38 ms, gap 6.44x
```

## P0：先拿到可见收益

目标：优先降低简单聚合查询延迟，并减少写入路径上的同步元数据落盘开销。

待办：

```text
1. [done] benchmark 输出阶段耗时：write / query / flush / compaction / recovery，方便后续按真实瓶颈排队。
2. [done] Manifest.UpdateIndexes 改成内存更新 + 延迟落盘，避免每批写入都重写 manifest.json。
3. [done] TsdbEngine 复用 segment metadata，避免连续查询反复整文件读取 segment 元数据。
4. [done] 简单聚合 fast path 继续收敛：mean/sum/count/min/max + 简单 tag filter 优先走 stats / buffer 累加，不回退到 ReadAllPoints。
5. [done] HTTP 对比脚本补输出 query report，确认真实压测中 aggregate pushdown 命中率。
6. [done] 内存 buffer 增加按 series 的轻量索引，tag filter 查询直接扫候选 series。
7. [done] 根据 query report 判断下一刀：当前聚合已命中 pushdown，且内部执行约 0-1 ms；segment metadata sidecar / footer 下沉到 P1。
```

验收：

```text
benchmark --format text/json/prometheus 能看到阶段耗时。
写入后内存索引立即可查；Dispose / Flush 后重新加载 manifest 仍可查。
重复执行简单聚合查询时不重复解析同一个 segment metadata。
现有聚合下推测试仍通过。
```

P0 后压测记录：

```text
测试时间：2026-06-28
口径：20,000 points, batch size 2,000, concurrency 1, warmup 后计时
MiniInflux：write 32,502.22 pts/s, aggregate 14.18 ms, raw LIMIT 1000 15.64 ms
InfluxDB 1.7.9：write 77,447.22 pts/s, aggregate 2.25 ms, raw LIMIT 1000 8.65 ms
MiniInflux query report：aggregate scanned 1,250 points, UsedAggregatePushdown=true, UsedSeriesIndexPushdown=true, DurationMs=0
```

写入专项复测记录：

```text
测试时间：2026-06-28
口径：20,000 points, batch size 2,000, concurrency 1；100,000 points, batch size 5,000, concurrency 1
本轮尝试：parser simple path 减少行切片、写入批内去重快路、schema 批内字段汇总、manifest/series 批内去重、buffer replay floor 增量维护。
结果：压测脚本隔离同步 flush 后，100,000 点 MiniInflux write 50,366.88 pts/s，InfluxDB 1.7.9 write 90,251.71 pts/s，同轮差距约 1.79x，已进入“差距 1 倍以内”口径。
结论：之前 100,000 点 /write 300s timeout 来自默认 50,000 点同步 flush 污染压测；下一步 P0 应转向后台 flush 或写入列式 batch，继续减少每点 Point+Dictionary+字符串重建。
```

## P1：查询主链路优化

目标：把聚合查询从“点物化后计算”推进到“列/块统计流式计算”。

待办：

```text
1. 为 segment 增加独立 metadata footer 或 sidecar，查询先读小块 metadata，而不是 File.ReadAllBytes 整文件。
2. ReadAllPoints / EnumeratePoints 增加流式聚合入口，避免简单聚合先 Rebuild 全部 Point。
3. GROUP BY tag / time 优先使用 series + column stats 分组累加，减少 Dictionary<(measurement,tags,time), Point> 重建。
4. BuildValuePairs / Calc 路径减少重复 OrderBy / ToList，单次有序扫描完成常见函数。
```

验收：

```text
aggregate query latency 从 175 ms 级别降到 15 ms 以内。
raw LIMIT 1000 query latency 降到 20 ms 以内。
查询报告能区分 stats pushdown、buffer scan、point materialization 三种路径。
```

## P2：结构性优化

目标：接近 InfluxDB 1.x 的单机常见 workload 表现，但不急着重写存储引擎。

待办：

```text
1. buffer 按 measurement + series 分桶，查询直接缩小扫描范围。
2. 索引补齐 measurement -> series -> shard/segment 映射，tag filter 后直接缩小 segment 集合。
3. 评估持久化倒排索引、列块级 min/max、后台预聚合。
4. 增加高 cardinality、多 measurement、乱序写入、删除 + compaction 混合 workload。
```

验收：

```text
write throughput 达到 80k-100k pts/s 区间。
aggregate query latency 进入 5-10 ms 区间。
优化后仍保持 Native AOT 友好，不引入重量级依赖。
```
