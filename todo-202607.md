# MiniInflux.Net10 2026-07 性能优化清单

更新时间：2026-07-03

待办：

```text

4. [done] 简单聚合 fast path 继续收敛：mean/sum/count/min/max + 简单 tag filter 优先走 stats / buffer 累加，不回退到 ReadAllPoints。
5. [done] HTTP 对比脚本补输出 query report，确认真实压测中 aggregate pushdown 命中率。
6. [done] 内存 buffer 增加按 series 的轻量索引，tag filter 查询直接扫候选 series。
7. [done] 根据 query report 判断下一刀：当前聚合已命中 pushdown，且内部执行约 0-1 ms；segment metadata sidecar / footer 下沉到 P1。
```

写入专项复测记录：

```text
测试时间：2026-06-28
口径：20,000 points, batch size 2,000, concurrency 1；100,000 points, batch size 5,000, concurrency 1
本轮尝试：parser simple path 减少行切片、写入批内去重快路、schema 批内字段汇总、manifest/series 批内去重、buffer replay floor 增量维护。
结果：压测脚本隔离同步 flush 后，100,000 点 MiniInflux write 89,660.11 pts/s，InfluxDB 1.7.9 write 114,307.84 pts/s，同轮差距约 1.27x，已进入“差距 1 倍以内”口径；raw LIMIT 1000 已做 buffer-only 单 series 倒序 LIMIT 下推，debug report 从扫描 6,250 points 降到 1,000 points，内部 DurationMs=4，wall time 最佳降到 26.34 ms。
结论：之前 100,000 点 /write 300s timeout 来自默认 50,000 点同步 flush 污染压测；下一步 P0 应转向后台 flush 或写入列式 batch，继续减少每点 Point+Dictionary+字符串重建。
raw 查询本轮推进：普通 /query 增加 buffer-only raw DESC LIMIT 专用 JSON 输出，绕过 QueryResponse Values 对象图；fast JSON 将固定 series tag 写入 Tags，不再逐行重复 tag 列。100,000 点复测中 raw 响应体从 58,519 bytes 降到 42,532 bytes，wall time 最佳 23.33 ms；debug 内部执行仍稳定在扫描 1,000 points / DurationMs=3 左右。
网络查证后修正方向：InfluxDB v1 官方 /query 支持 epoch 参数，默认 RFC3339，epoch=ns/u/ms/s/m/h 可返回 Unix epoch；压测脚本已补 -Epoch 和 5 次 query median。median 口径下 raw LIMIT 1000 默认模式 MiniInflux 5.92 ms、InfluxDB 9.09 ms；epoch=ns 模式 MiniInflux 2.97 ms、InfluxDB 7.84 ms。结论：raw 查询暂时不是最优继续投入点，后续 P0 优先回到写入吞吐。
写入本轮推进：参考 InfluxDB v1 WAL/cache 批量写入方向和 parser 低分配思路，WAL 改为同一 HTTP batch 写一条多行 record；/write 入口直接写 engine，去掉 Channel/TCS/worker 同步等待绕路，并移除不再使用的 WriteQueue 启动注册；未开启 MaxBufferPoints 时不再每批扫描全部 buffer；无重复点写入直接复用 parser 返回的 point list；series cardinality 改用 SeriesKey，去掉每点 record ToString 分配；simple parser 去掉 tag/field 计数预扫描和整批精确行数预扫描，并在单次 batch 内复用重复 measurement/tag/field 字符串。100,000 点 median 收口复测：MiniInflux write 120,418.14 pts/s，InfluxDB 159,606.27 pts/s，差距约 1.33x；最好一轮 MiniInflux write 132,987.99 pts/s，InfluxDB 167,168.70 pts/s。
写入收尾：本轮先落地 raw key/field 方向中兼容现有 Point 模型的低风险部分：simple parser 对同一 HTTP batch 内重复 series 复用已解析 tags 字典和 canonical tag key；写入内存路径把 buffer append、buffer series index、全局 series keys 和 manifest index 更新合成一趟，并且 manifest 只处理本批唯一 series。
收尾复测：同口径 100,000 points / batch size 5,000 / concurrency 1 / query median 5 次，MiniInflux write 133,971.07 pts/s，aggregate 4.53 ms，raw LIMIT 1000 5.95 ms；InfluxDB 1.7.9 本轮 write 94,931.71 pts/s，aggregate 30.67 ms，raw 9.39 ms。本轮 InfluxDB 波动偏低，结论按 MiniInflux 自身已稳定推进到 13 万 pts/s 级别记录。
写入下一步：近期不再继续小刀式优化；若要再明显推进，应做更接近 InfluxDB 的 raw key/field point 模型或列式 batch 写入入口，例如让 parser 先产出不可变 raw series key + raw field slice，再由 schema/WAL/cache 按需解析字段。
2026-07-04 修正压测口径：默认压测不再只测 WAL/buffer，会在 MiniInflux 写入计时后显式 flush 并输出 FlushAfterWriteMs / SegmentFiles；同时 `/write` 默认不再同步触发 segment flush，改为 WAL+buffer 返回、后台/显式 flush 落盘，更接近 InfluxDB WAL/cache 写入模型。本轮 100,000 points / batch 5,000 / concurrency 1 / query median 3 次：MiniInflux write 87,115.00 pts/s，FlushAfterWriteMs 2,246.67，aggregate 1.93 ms，raw 12.89 ms，SegmentFiles=1；InfluxDB 1.7.9 write 65,281.32 pts/s，aggregate 11.09 ms，raw 21.19 ms。
```

## 2026-07-03 全局代码扫描结论

扫描范围：`Program.cs`、`Query/`、`Storage/`、`Protocol/`、`Model/`、管理 CLI 和现有测试。重点搜索全量读、点物化、重复排序/分组、锁内 IO、索引扫描、compaction 重建等明显性能风险。

优先级结论：

```text
P0-1. [done] 普通 raw SELECT 在 buffer-only 安全场景复用 streaming scan；简单 `sum/count/min/max/mean + GROUP BY time/tag` 已走 streaming aggregate 边读边累加。复杂函数如 derivative/top/sample/stddev/percentile 仍需要完整序列语义，跨 segment raw SELECT 也需要全局排序/去重，保留旧路径避免语义回归。
P0-2. [done] TsdbEngine.ReadAllPoints / EnumeratePoints 的 segment 读取、tombstone 过滤逻辑已抽成共享 ReadSegmentColumns；raw buffer-only 和简单聚合已避开全量 Point 物化，跨 segment/复杂函数保留旧的 Rebuild -> OrderBy -> Deduplicate 作为正确性路径。
P0-3. [done] SegmentReader.ReadMetadata 已支持 v4 footer 快读；QueryExecutionReport 已暴露 SegmentMetadataFooterHits / SegmentMetadataFullReads，并补 flush 后 segment 聚合测试确认 footer 命中。

P1-1. [done] Compactor 已增加列直写路径：MergeColumns 后直接 SegmentWriter.WriteColumns，并用列级 schema/index 更新，避免 ColumnsToPoints -> WriteSegment 再 GroupBy。
P1-2. [done] GROUP BY tag pushdown 已增加 canonical tag 批内解析缓存，避免同 series 多字段 metadata 反复 ParseTagsCanonical。
P1-3. [done] SchemaRegistry 已从 "db|measurement|field" 字符串 key 改成 db -> measurement -> fields 嵌套字典，字段数检查不再 StartsWith 全表扫描，GetFields/ListMeasurements 不再 Split。
P1-4. [done] ShardManager.ListSegments 已优先使用 manifest SegmentFiles 生成路径，在线查询不再每次 Directory.GetFiles 扫 shard 目录；孤儿 segment 留给 validate/repair 处理。
P1-5. [done] TombstoneStore.FilterColumnDeleted 已将匹配 tombstone 合并成时间区间，列过滤从每点 Any(tombstones) 改为按有序区间线性推进。

P2-1. [done] Manifest tag/series 查询已改为锁内快照、锁外 Distinct/Order/Regex，降低索引读查询对写入更新 manifest 的锁占用。
P2-2. [done] ListTagKeys/ListTagValues 已只依赖 manifest/tag index，不再 fallback 到 ReadAllPoints 全库扫描；索引不完整时由 repair/recover 负责重建。
P2-3. 写入若继续追 InfluxDB 级别，才考虑 raw key/field point 或列式 batch 入口；当前不是下一刀，除非压测证明 P0 查询侧已收敛。
P2-4. 管理 CLI / inspect / validate 可以接受全量扫描，不进入性能 P0；只需避免影响在线 query/write 热路径。
```

## P1：查询主链路优化

目标：把聚合查询从“点物化后计算”推进到“列/块统计流式计算”。

待办：

```text
1. [done] 为 segment 增加独立 metadata footer，查询先读小块 metadata，而不是 File.ReadAllBytes 整文件；旧 v1/v2/v3 segment 保留全量解析兼容。
2. [done] ReadAllPoints / EnumeratePoints 已提供简单流式聚合入口，避免常见简单聚合先 Rebuild 全部 Point。
3. [done] GROUP BY tag / time 的简单 sum/count/min/max/mean 已边读边累加，减少 Dictionary<(measurement,tags,time), Point> 重建。
4. [done] BuildValuePairs / Calc 的常见简单函数已由 streaming aggregate 绕开；复杂序列函数保留旧路径以保证语义。
```

验收：

```text
aggregate query latency 从 175 ms 级别降到 15 ms 以内。
raw LIMIT 1000 query latency 降到 20 ms 以内。
查询报告能区分 stats pushdown、buffer scan、point materialization 三种路径。
```

## P2：结构性优化

目标：接近 InfluxDB 1.x 的单机常见 workload 表现，但不急着重写存储引擎。

待办：

```text
1. buffer 按 measurement + series 分桶，查询直接缩小扫描范围。
2. 索引补齐 measurement -> series -> shard/segment 映射，tag filter 后直接缩小 segment 集合。
3. 评估持久化倒排索引、列块级 min/max、后台预聚合。
4. 增加高 cardinality、多 measurement、乱序写入、删除 + compaction 混合 workload。
```

验收：

```text
write throughput 达到 80k-100k pts/s 区间。
aggregate query latency 进入 5-10 ms 区间。
优化后仍保持 Native AOT 友好，不引入重量级依赖。
```
